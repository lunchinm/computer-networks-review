5.1 运输层概述

本章核心问题：如何为运行在不同主机上的应用进程提供端到端的逻辑通信？

运输层的功能 重点

  • 复用与分用——多个应用进程共享同一运输层
  • 端口号——标识应用进程：服务器端用熟知端口（0~1023），客户端用短暂端口（49152~65535）
  • 可靠传输——TCP 提供（确认+重传+序号）

UDP 与 TCP 对比 必考

特性UDPTCP
连接无连接面向连接
可靠性尽最大努力（不可靠）可靠，保证按序到达
开销首部 8 字节首部 20~60 字节
流量/拥塞控制无有（滑动窗口/拥塞窗口）
适用场景实时音视频、DNS、广播文件传输、Web、邮件
数据单位UDP 用户数据报TCP 报文段
易错：UDP 不是「不可靠就一定不好」——UDP 无连接、开销小、实时性好，非常适合实时音视频。TCP 有拥塞控制反而会导致音视频卡顿。

5.2 UDP 协议

UDP 的特点 重点

  • 无连接——发送前不需建立连接
  • 尽最大努力交付——不保证可靠交付
  • 面向报文——对应用层交下来的报文不合并也不拆分，保留边界
  • 没有拥塞控制——网络拥塞不影响发送速率
  • 支持一对一、一对多、多对多通信
  • 首部仅 8 字节：源端口(2)+目的端口(2)+长度(2)+检验和(2)

5.3 TCP 协议

TCP 首部格式 重点

源端口 + 目的端口 + 序号（32 位） + 确认号（32 位） + 首部长度 + 标志位（URG/ACK/PSH/RST/SYN/FIN）+ 窗口大小 + 检验和 + 紧急指针 + 选项。

标志位含义
SYN同步序号，用于建立连接
ACK确认号字段有效
FIN释放连接
RST重置连接
PSH推送，不等待缓存填满就交付

TCP 可靠传输的实现 必考

以字节为单位的滑动窗口：

  • 滑动窗口以字节为单位
  • 发送窗口 = 已发送未确认 + 可发送但未发送
  • 窗口前沿 = 序号 + 窗口大小（由接收方告知）
  • 收到确认后窗口前移（滑动）

超时重传时间的选择 必考

问题：超时重传时间 RTO 设太短→虚假重传；太长→恢复慢。

解决：自适应算法——根据 RTT 样本计算加权平均往返时间 RTTs。

$$RTT_S = (1 - \\alpha) \\times RTT_S + \\alpha \\times \\text{新 RTT 样本}$$

$$RTO = RTT_S + 4 \\times RTT_D$$（RFC 6298 推荐 $\\alpha = 1/8$）

易错：计算题中，RTO 不是简单的 2×RTT，需要考虑 RTT 的偏差 RTT_D。

TCP 流量控制 重点

流量控制——控制发送方的发送速率，使其不超过接收方的接收能力（由接收方窗口 rwnd 控制）。

TCP 通过报文中窗口字段告诉对方自己还能接收多少字节（缓存剩余空间）。

死锁问题：接收方窗口=0 时停止发送，之后窗口变大但通知丢失。解决方案：持续计时器——发送方定期发送探测报文。

TCP 拥塞控制 必考

拥塞控制 vs 流量控制：拥塞控制防止过多的数据注入网络（全局问题），流量控制是端到端的速率匹配（点对点）。

四种算法：

算法阶段cwnd 变化
慢开始启动或超时后cwnd 从 1 开始，每收到 ACK 翻倍（指数增长）
拥塞避免cwnd ≥ ssthresh每 RTT 线性加 1（加法增大）
快重传收到 3 个重复 ACK立即重传缺失的报文段
快恢复快重传后ssthresh = cwnd/2，cwnd = ssthresh，进入拥塞避免

乘法减小 AIMD：发生超时→ssthresh = cwnd/2，cwnd = 1，重新慢开始。

易错：区分快恢复和超时——收到 3 个重复 ACK（快重传+快恢复，cwnd 减半），超时（慢开始，cwnd 重置为 1）。两个处理严重程度不同。

TCP 连接管理 必考

三次握手：

  1. 客户端：SYN=1, seq=x → 服务器（SYN-SENT）
  2. 服务器：SYN=1, ACK=1, seq=y, ack=x+1 → 客户端（SYN-RCVD）
  3. 客户端：ACK=1, seq=x+1, ack=y+1 → 服务器（ESTABLISHED）

为什么要三次？防止已失效的连接请求突然传送到服务器，导致错误——如果只有两次握手，服务器一旦发出确认就建立连接，此时若客户端未响应，服务器资源被浪费。

四次挥手：

  1. 客户端：FIN=1, seq=u → 服务器（FIN-WAIT-1）
  2. 服务器：ACK=1, seq=v, ack=u+1 → 客户端（CLOSE-WAIT，客户端收到后 FIN-WAIT-2）
  3. 服务器：FIN=1, ACK=1, seq=w, ack=u+1 → 客户端（LAST-ACK）
  4. 客户端：ACK=1, seq=u+1, ack=w+1 → 服务器（TIME-WAIT → CLOSED）

TIME-WAIT：客户端发完最后 ACK 后等待 2MSL 时间，确保服务器收到 ACK。
